# SpringCloud
## 核心概念
Zuul = 大门门卫（网关，统一入口）
字面：祖尔（印度神话里地狱守门恶鬼、门禁守护者）
命名寓意：网关作为请求的大门，拦截、管控所有流量。

Sentinel = 巡逻哨兵（实时流量监控防护）
字面：哨兵、守卫、岗哨
命名寓意：像哨兵实时监控接口流量，一旦异常立刻拦截限流、熔断防护。

Hystrix = 针对 “系统崩溃” 问题的防护工具
词根来自 hysteria，字面：歇斯底里、失控崩溃
命名寓意：组件用来防止服务调用链 “失控雪崩”，阻止系统陷入崩溃。

Ribbon：丝带 → 串联服务做负载均衡
字面：丝带、缎带、带状捆束
命名寓意：把多个服务实例 “串联绑定” 起来，做客户端负载均衡，像丝带一样串联所有可用节点。

OpenFeign：模拟伪装 → 伪装成本地接口远程调用
Feign：假装、伪装、模拟
Open：开源开放
命名寓意：伪装成本地接口，开发者像调用本地方法一样远程调用服务，屏蔽底层 HTTP 网络细节。

Kubernetes：船长 / 舵手 → 容器集群调度管理者
词源来自希腊语 κυβερνήτης，字面：舵手、领航员、船长
命名寓意：掌控、调度所有容器，像船长指挥整艘船，统一管理集群内所有微服务容器。
行业小知识：中间 8 个字母 ubernetes 省略，简写 K8s。

### 一、Zuul（Spring Cloud 网关，你的强项）
#### 核心定位
Spring Cloud 第一代API网关，基于**同步阻塞Servlet**实现，负责请求统一入口、路由分发、鉴权、限流、日志、灰度发布。
#### 核心机制
1. 四大过滤器生命周期：`PRE`前置、`ROUTE`路由转发、`POST`响应处理、`ERROR`异常拦截，可自定义业务逻辑。
2. 整合了Ribbon做服务负载均衡，整合Hystrix实现路由熔断。
3. 缺陷：同步阻塞模型，Tomcat线程池有限，高并发场景易线程耗尽、吞吐量低，**Spring Cloud官方已停止迭代，推荐Gateway替代**。
#### 适用场景
老旧微服务架构兼容、简单轻量网关需求；高并发、大流量项目不推荐。

### 二、Spring Cloud LoadBalancer（客户端负载均衡，你的强项）
#### 核心定位
Spring Cloud官方推出，**Ribbon替代方案**，客户端侧负载均衡组件，脱离Netflix体系，原生适配SpringBoot3/新版云组件。
#### 核心机制
1. 从注册中心（Nacos/Eureka）拉取服务实例列表，本地内存缓存。
2. 内置轮询、随机策略，支持**全局/单服务自定义均衡规则**（本地优先、权重、最小连接数）。
3. 无缝适配OpenFeign、RestTemplate，无需额外改造；无独立线程池，轻量化性能损耗较小。
#### 和Ribbon关键区别
Ribbon属于Netflix闭源组件，停止维护；LoadBalancer是Spring官方自研，持续更新、适配云原生K8s。

### 三、Hystrix（服务熔断降级，你的弱项）
#### 核心定位
Netflix开源分布式服务容错组件，通过**线程池隔离+滑动窗口熔断**解决微服务调用雪崩问题。
#### 核心机制
1. 线程池隔离：不同下游服务分配独立线程池，一个服务故障不会拖垮全部调用。
2. 熔断三状态：关闭（正常放行）→打开（直接走降级逻辑）→半开（少量试探请求自动恢复）。
3. 滑动窗口统计超时、异常比例，超过阈值触发熔断；通过`@HystrixCommand`注解实现降级方法。
#### 致命短板
1. 基于AOP代理，同类`this`自调用注解失效；
2. 项目早已停止维护，无新特性、不支持云原生集群限流；
3. 能力单一，仅熔断降级，无精细化热点限流、实时监控面板。

### 四、Sentinel（流量防护，你的弱项）
#### 核心定位
阿里开源新一代分布式流量控制组件，全方位流量防护，是Hystrix生产级替代品。
#### 核心能力（远超Hystrix）
1. 多维度管控：QPS限流、线程数限流、**热点参数限流**、集群限流、链路限流；
2. 熔断降级：支持异常比例、慢调用比例、异常数三种熔断策略；
3. 自带可视化Dashboard控制台，实时监控接口流量、异常；
4. 规则动态推送，修改规则无需重启服务；支持SPI自定义限流逻辑。
#### 和Hystrix对比
Hystrix侧重故障隔离；Sentinel兼顾限流、熔断、系统自适应保护，云原生项目首选。

### 五、OpenFeign（声明式远程调用，你的弱项）
#### 核心定位
Spring Cloud声明式HTTP调用框架，封装HTTP请求，简化微服务之间远程接口调用。
#### 核心机制
1. 基于接口+注解定义远程API（`@FeignClient`指定服务名），无需手动拼接URL、封装HTTP工具；
2. 新版自动整合LoadBalancer实现客户端负载均衡；
3. **原生无熔断能力**，必须手动整合Hystrix/Sentinel才能实现降级；
4. 默认集成Jackson序列化，支持自定义编码器、解码器、拦截器。
#### 高频踩坑考点
1. 接口参数传递需注意`@RequestParam`/`@RequestBody`注解规范，否则参数丢失；
2. 同类内部调用Feign接口不会触发负载均衡；
3. 超时时间分Feign连接超时、读超时两层配置。

### 六、Kubernetes(K8s，容器编排，你的强项)
#### 核心定位
云原生标准容器编排平台，自动化管理Docker容器，实现微服务部署、扩缩容、自愈、滚动更新、资源调度。
#### 核心核心资源对象
1. Pod：最小调度单元，运行业务容器；本地存储临时数据，销毁丢失；持久化依赖PV/PVC。
2. Deployment：管理无状态微服务，实现滚动发布、副本扩缩容、故障自动重启。
3. Service：固定访问入口，屏蔽Pod动态IP，内置四层负载均衡，供网关/服务内部调用。
4. ConfigMap/Secret：统一管理配置文件、环境变量、加密密钥。
#### 微服务场景价值
1. 自动扩缩容：流量高峰自动增加Pod实例，低谷缩容节约资源；
2. 自愈：Pod异常崩溃，平台自动重建；
3. 灰度/蓝绿发布：零停机更新微服务版本；
4. 资源隔离：CPU、内存配额限制，避免服务资源抢占。

# Spring
### Spring 事务管理器
#### 题目
项目多数据源场景，分别配置 DataSource1、DataSource2，对应两个事务管理器txManager1、txManager2；在方法上使用@Transactional(value = "txManager2", rollbackFor = Exception.class)，下列说法错误的是？
A. value 指定当前方法使用 txManager2 管理事务
B. 若省略 value，会使用容器中默认标记@Primary的事务管理器
C. rollbackFor = Exception.class 时，除非捕获一场，否则所有受检、非受检异常都会触发回滚
D. 不配置 rollbackFor 时，捕获 Exception 不会触发事务回滚，仅 RuntimeException 触发
#### 正确答案
C
#### 简洁解析
A 正确：@Transactional 的 value 属性绑定指定事务管理器。
B 正确：多事务管理器必须指定 value，无指定则取 @Primary 主管理器。
C 错误：rollbackFor=Exception.class 抛出异常时回滚；若代码 try-catch 捕获 Exception，不会触发回滚。
D 正确：默认规则下 RuntimeException、Error 都会触发回滚，选项只写 RuntimeException。
#### 考点总结
- 单层事务、仅 try-catch 吞异常、无手动标记：不回滚；
#### 延伸考点
- catch 内调用setRollbackOnly()：捕获异常也回滚；
- REQUIRED 嵌套事务，子方法抛异常：外层捕获依然回滚；（因为内外层事务标记是共享的，一旦内层打上 rollback-only，整个事务再也无法提交）
- Spring 默认事务传播级别：Propagation.REQUIRED，默认隔离级别：Isolation.DEFAULT，使用底层数据库默认隔离级别；
- Spring 默认自动回滚范围：RuntimeException、Error。

### Spring 过滤器与拦截器
#### 题目
关于 Filter 过滤器与 SpringMVC Interceptor 拦截器执行顺序、底层容器归属，说法正确的是？
A. 拦截器先执行，过滤器后执行；二者都由Spring IoC容器管理
B. Filter属于Servlet规范，由Tomcat容器管理；Interceptor由DispatcherServlet创建，归属Spring IoC
C. 当 Interceptor 的 postHandle() 方法抛出异常时，后续 Interceptor 的 afterCompletion() 依然会正常执行，但 Filter 的后置逻辑不受影响
D. 过滤器无法注入Spring容器Bean，拦截器不能注入Servlet原生对象
#### 正确答案
**B**
#### 简洁解析
1. A 错误：执行顺序Filter → Interceptor；Filter不归IoC管理。
2. B 正确：Filter是Servlet API组件，由Web容器创建；拦截器是SpringMVC组件，由IoC管理。
3. C 错误：postHandle() 抛出异常时，当前 Interceptor 及后续所有 Interceptor 的 afterCompletion() 都不会执行，过滤器后置逻辑不受此影响。
4. D 错误：Filter可通过代理/注册Bean注入IoC，拦截器也可获取Servlet相关对象。
#### 考点总结
- Filter 与 Interceptor 分属两套体系：Filter 基于 Servlet 规范，由 Web 容器管理；Interceptor 基于 Spring MVC，由 Spring IoC 容器管理。
- 执行顺序为 Filter → DispatcherServlet → Interceptor，且异常传播路径不同，拦截器内部异常不影响 Filter 的后置逻辑。
- Filter 可通过注册为 Spring Bean 的方式注入容器中的依赖，但其生命周期仍由 Web 容器管理。

### Filter、Interceptor、@ControllerAdvice 异常场景执行顺序
#### 题目
同一个请求链路中，自定义Filter、自定义HandlerInterceptor、@ControllerAdvice异常处理执行顺序，下列流程正确的是？

B. Filter前置 → Interceptor preHandle → Controller报错 → @ControllerAdvice → Interceptor afterCompletion → Filter后置

### Spring Ioc父子容器模型
#### 题目
基于Spring IoC父子容器模型，关于DispatcherServlet创建的子容器，说法错误的是？
A. 子容器可直接获取父IoC容器中注册的Service、Dao Bean
B. 根容器无法访问MVC子容器内的@Controller、拦截器Bean
C. 拦截器注册在MVC子容器，因此拦截器内无法注入根容器的Service
D. 多DispatcherServlet场景，每个Servlet拥有独立MVC子容器，互不共享控制器Bean
#### 正确答案
**C**
#### 简洁解析
1. A 正确：子容器可向上查找父容器Bean。
2. B 正确：双亲委派模型单向查找，父容器看不到子容器Bean。

### 拦截器 preHandle 返回 false 的执行链路分析
#### 题目
开发自定义拦截器，重写preHandle返回false，下列现象描述正确的是？
A. 控制器方法执行，不执行postHandle，直接执行afterCompletion
B. DispatcherServlet不再继续后续处理器流程，不会进入Controller，afterCompletion依旧执行
C. 所有Filter后置逻辑会被跳过，直接返回响应
D. preHandle返回false时，全局@ControllerAdvice异常处理器会捕获拦截器阻断异常
#### 正确答案
**B**
#### 简洁解析
1. A 错误：preHandle返回false直接截断，控制器完全不执行。
2. B 正确：preHandle=false中断链路，跳过Controller与postHandle，但一定会执行afterCompletion做资源清理。
3. C 错误：Filter是外层组件，后置逻辑不受拦截器返回值影响，正常执行。
4. D 错误：preHandle主动阻断不属于异常，不会进入全局异常处理器。
#### 考点总结
拦截器preHandle返回false的链路截断规则，区分postHandle、afterCompletion执行时机。
#### 参考源码
```java
boolean applyPreHandle(HttpServletRequest request, HttpServletResponse response) throws Exception {
    HandlerInterceptor[] interceptors = getInterceptors();
    if (!ObjectUtils.isEmpty(interceptors)) {
        for (int i = 0; i < interceptors.length; i++) {
            HandlerInterceptor interceptor = interceptors[i];
            // 1. 正序执行每个拦截器的 preHandle
            if (!interceptor.preHandle(request, response, this.handler)) {
                // 2. 一旦有拦截器返回 false，立即执行清理工作
                triggerAfterCompletion(request, response, null);
                // 3. 返回 false，通知 DispatcherServlet 中断流程
                return false;
            }
            // 4. 记录最后一个成功执行的拦截器的索引
            this.interceptorIndex = i;
        }
    }
    return true;
}
```

假设我们要对分布式SQL进行优化，下列方法无效的是？

### WHERE与HAVING的执行时机
#### 题目
关于WHERE与HAVING子句的执行时机、可用过滤字段，下列说法错误的是？
A. WHERE 在分组 GROUP BY 前过滤原始数据，不能使用聚合函数；HAVING 在分组后过滤分组结果，仅支持聚合字段
B. 查询语句 `select id,sum(score) s from student where s>60 group by id` 语法合法，可正常执行
C. HAVING 可以使用SELECT中定义的聚合别名，MySQL支持，标准SQL部分数据库不支持
D. 若需求为“过滤总分大于300的学生分组”，过滤条件必须写在HAVING，不能写WHERE
#### 正确答案
**B**
#### 简洁解析
1. A 正确：执行顺序：FROM→WHERE→GROUP BY→聚合计算→HAVING→ORDER BY；WHERE无法识别聚合结果。
2. B 错误：s是聚合别名，WHERE阶段还未完成sum计算，无法识别，直接报语法错误。
3. C 正确：MySQL解析顺序特殊，允许HAVING引用聚合别名；Oracle等标准数据库不允许。
4. D 正确：总分是分组聚合后结果，WHERE无法过滤，只能用HAVING。
#### 考点总结
WHERE与HAVING执行顺序差异，聚合函数不能在WHERE中使用，HAVING专用于分组后过滤。

FROM   →   WHERE   →   GROUP BY   →   聚合计算   →   HAVING   →   ORDER BY
↑            ↑            ↑            ↑            ↑            ↑
表名      原始行过滤     分组           sum/avg     分组后过滤    排序
❌ 不能用别名   ✅ 可用别名   ✅ 可用别名   ✅ 可用别名   ✅ 可用别名

#### 题目
数据表 `score`，主键自增 `id`，字段 `class_id`、`score`；需求：统计每个班级平均分大于 80 的班级，按平均分降序。下列 SQL **语法错误**的是？
A. `SELECT class_id, AVG(score) avg_s FROM score GROUP BY class_id HAVING AVG(score) > 80 ORDER BY avg_s DESC`  
B. `SELECT class_id, AVG(score) avg_s FROM score WHERE score > 0 GROUP BY class_id HAVING avg_s > 80 ORDER BY avg_s DESC`  
C. `SELECT class_id, AVG(score) avg_s FROM score GROUP BY class_id ORDER BY avg_s DESC HAVING avg_s > 80`
D. `SELECT class_id, AVG(score) avg_s FROM score GROUP BY class_id HAVING avg_s > 80 ORDER BY avg_s DESC`

#### 正确答案
**C**

#### 简洁解析
1. **A、B、D 正确**：遵循 `GROUP BY → HAVING → ORDER BY` 的语法顺序，可正常执行。
2. **C 错误**：`HAVING` 必须位于 `GROUP BY` 之后、`ORDER BY` 之前；该语句将 `HAVING` 写在 `ORDER BY` 之后，违反 SQL 语法规则，**直接报错**。

#### 考点总结
`GROUP BY`、`HAVING`、`ORDER BY` 的固定书写顺序：`GROUP BY` → `HAVING` → `ORDER BY`，`HAVING` 必须位于分组后、排序前。

#### 题目
关于自增主键、分组HAVING、排序组合场景，描述正确的是？
A. 自增主键连续无空缺，GROUP BY分组后按自增id排序一定有序，可省去order by
B. 表删除部分数据后自增主键断裂，不影响HAVING聚合统计结果
C. HAVING中使用自增主键id做过滤条件，可直接利用主键索引快速过滤分组
D. 使用自增主键分组，HAVING过滤效率高于普通字段分组，因为主键自带索引

**B**

### 索引的生效范围
#### 题目
一张用户表 `user`，主键 `id` 为自增 INT，普通索引 `idx_phone(phone)`，执行 SQL：
`SELECT name FROM user WHERE phone = '13800001111' GROUP BY phone HAVING COUNT(*) > 1 ORDER BY id DESC;`

关于该 SQL 的执行逻辑与优化分析，下列说法正确的是？
A. `HAVING COUNT(*) > 1` 会在分组前过滤掉不满足条件的记录，从而减少 `GROUP BY` 的分组数据量，提升索引效率  
B. 索引 `idx_phone` 可同时覆盖 `WHERE` 等值过滤和 `GROUP BY` 分组，但 `HAVING` 聚合过滤和 `ORDER BY id DESC` 排序仍会产生临时表或文件排序  
C. 将 `HAVING COUNT(*) > 1` 改写为 `WHERE COUNT(*) > 1`，可在分组前提前过滤数据，避免聚合计算  
D. 主键 `id` 为自增 INT，`GROUP BY phone` 分组后的结果集仍按 `id` 保持有序，因此 `ORDER BY id DESC` 可直接复用主键索引顺序，无需额外排序  

#### 正确答案
**B**

#### 简洁解析  
1. **A 错误**：`HAVING` 的执行顺序在 `GROUP BY` 和聚合计算（`COUNT(*)`）**之后**，只能过滤分组后的结果，**无法减少分组数据量**。  
2. **B 正确**：`idx_phone(phone)` 可覆盖 `WHERE phone = ?` 过滤和 `GROUP BY phone` 分组；但 `HAVING` 聚合过滤和 `ORDER BY id DESC` 排序需要额外临时表/文件排序。
3. **C 错误**：`COUNT(*)` 是聚合函数，`WHERE` 阶段尚未执行分组和聚合，**语法不合法**，无法提前过滤。
4. **D 错误**：`GROUP BY phone` 会按 `phone` 重新组织数据，结果集顺序与 `id` 无关，`ORDER BY id DESC` 需重新排序，**无法直接利用主键索引有序性**。

#### 考点总结
- `HAVING` 与 `WHERE` 执行顺序差异：`WHERE` 过滤原始行 → `GROUP BY` → 聚合计算 → `HAVING` 过滤分组结果
- 索引只能优化 `WHERE` 和 `GROUP BY`，无法优化 `HAVING` 聚合过滤
- `GROUP BY` 打乱结果集顺序，`ORDER BY` 需独立排序，不能依赖主键自增特性

#### 题目
表student，主键自增id，唯一索引uk_card(card_no)，联合索引idx_class_score(class,score)；执行语句：
`SELECT class,MAX(score) max_s FROM student WHERE class IN (1,2) GROUP BY class HAVING MAX(score)>=90 ORDER BY max_s DESC;`
下列说法错误的是？
A. idx_class_score可覆盖where过滤、group by分组、max聚合，减少回表
B. HAVING基于聚合结果过滤，无法利用索引提前过滤数据
C. 若删除联合索引，仅保留主键索引，该SQL会产生临时表+文件排序
D. 将HAVING条件移至WHERE中，可直接走索引过滤，大幅提升性能
#### 正确答案
**D**
#### 简洁解析
1. A 正确：联合索引前缀class用于筛选分组，score字段支持聚合覆盖。
   联合索引 idx_class_score (class, score)
   │                               │
   ▼                               ▼
   用于 WHERE class IN (1,2)  用于 MAX(score)
   用于 GROUP BY class        聚合值直接从索引取
2. B 正确：聚合计算完成后才执行HAVING，索引无法提前拦截不满足聚合条件的数据。
3. C 正确：无联合索引时，分组、聚合、排序都会生成临时文件，性能暴跌。
4. D 错误：MAX(score)是聚合函数，WHERE阶段不存在聚合结果，语法不支持迁移。
#### 考点总结
索引仅优化原始行过滤，HAVING聚合过滤无法下推到WHERE，结合联合索引覆盖考点。
