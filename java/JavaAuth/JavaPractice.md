# Kafka
### Zookeeper 管理的元数据
#### 题目
以下哪项不属于Zookeeper管理的Kafka元数据？
A. Controller选举信息，包括当前Controller Broker ID
B. Broker注册信息，包括Broker ID和网络地址
C. 消费者组的offset位移信息
D. Topic配置信息，包括分区数量和副本分配

#### 答案
**C**

#### 精简解析
1. **A 属于ZK管理**
   Controller主节点选举信息持久化在ZK，集群依靠ZK完成主节点切换。

2. **B 属于ZK管理**
   Broker启动后在ZK注册节点，记录ID、IP端口，下线节点自动清除。

3. **C 不属于ZK管理**
   Kafka 0.9及以上版本，消费offset统一存内置主题`__consumer_offsets`，不再保存在ZK。

4. **D 属于ZK管理**
   Topic分区、副本数量与分配方案全部记录在ZK中。

#### 考点小结
传统ZK架构下，集群、Broker、Topic、Controller元数据由ZK维护；消费者偏移量由Kafka内部主题保存。

### Zookeeper 重新选举
#### 题目
Kafka在高可用环境中使用Zookeeper来管理集群状态。当Kafka集群中的控制器节点故障时，以下哪个过程会在Zookeeper的帮助下触发，以确保集群恢复正常？
A. Kafka所有Broker自动重启，重新加入集群
B. 由Zookeeper重新选举一个新的控制器节点来承担控制器角色
C. Zookeeper将所有现有分区的领导者切換到不同的Broker，以确保消息可用
D. Kafka生产者和消费者立即停止所有消息操作，等待手动恢复控制器

#### 正确答案
**B**

#### 简洁解析
1. **A错误**
   控制器故障不会让全部Broker重启，无该机制。
2. **B正确**
   控制器在ZK抢占临时节点，原控制器宕机后临时节点消失，其他Broker竞争创建节点，完成新控制器选举。
3. **C错误**
   分区leader切换是新控制器上线后执行的操作，不是ZK直接完成。
4. **D错误**
   集群会自动选新控制器，无需手动恢复，生产消费不会全部暂停。

#### 考点总结
ZK通过临时节点实现Kafka控制器自动重新选举，保障集群管理功能正常。

## Redis
### Redis 基本数据类型
#### 题目
常用中间件Redis的基本数据类型不包括？
A. Int
B. List
C. String
D. Hash

#### 正确答案
**A**

#### 简洁解析
1. A（Int）：Redis没有单独的Int基础类型，数字本质存为String，不属于基础数据类型。
2. B（List）：Redis基础类型，有序链表。
3. C（String）：Redis最基础的数据类型。
4. D（Hash）：Redis基础类型，类似map结构。

#### 考点总结
Redis五大基础类型：String、List、Set、ZSet、Hash，不存在Int类型。

## SQL
### SQL 优化
#### 题目
假设我们要对分布式SQL进行优化，下列方法无效的是？
A. 可以使用 union all 的情况下，不要使用 union
B. 两表join 时，条件字段尽量放在where 里面
C. 避免关联字段分布倾斜
D. 大表join 小表时，可以考虑 map-side join

#### 正确答案
**B**

#### 简洁解析
1. A 有效：`union` 会去重排序，`union all` 直接拼接无开销，优先使用。
2. B 无效：多表关联过滤条件建议写在 `on` 后，放where会提前过滤影响关联逻辑，无法优化。
3. C 有效：关联字段数据倾斜会造成任务卡死，提前打散可优化执行效率。
4. D 有效：小表广播到Map端做关联，减少Shuffle开销，适合大表关联小表场景。

#### 考点总结
分布式SQL关联查询，过滤条件区分`on`与`where`，关联条件写on更合理。

### SQL Session
#### 题目
对于MyBatis一级缓存说法错误的是？
A. 同一个SqlSession只能开启一级缓存
B. 不同的SqlSession，即使同一个Mapper也会到数据库中进行不同的查询，即不同的SqlSession一级缓存是无效的
C. 在参数与SQL完全一样的情况下并且不声明刷新缓存没超时的，使用同一个SqlSession对象调用同一个Mapper方法时，SqlSession只会取出当前缓存数据，不会再到数据库中进行查询
D. 在默认没有配置的情况下，只针对同一个SqlSession而言只会开启一级缓存

#### 正确答案
**A**

#### 简洁解析
1. A 错误：同一个 SqlSession 既可以使用一级缓存（默认开启），也可以访问和提交由 Mapper 管理的二级缓存跨（SqlSession 共享）。
2. B 正确：一级缓存隔离在各自SqlSession，跨Session无法复用缓存，会重新查库。
3. C 正确：同会话、相同SQL参数、缓存未失效，直接读取缓存，不访问数据库。
4. D 正确：MyBatis默认开启一级缓存，作用范围仅限单个SqlSession。

#### 考点总结
MyBatis一级缓存默认开启，作用域为单个SqlSession，每个SqlSession缓存相互独立。

## 微服务
### Eureka自我保护机制
#### 题目
关于Eureka自我保护机制，说法错误的是？
A. 固定时间内大量实例被注销，可能会严重威胁整个微服务架构的可用性。为了解决这个问题，Eureka 开发了自我保护机制
B. 自我保护模式正是一种针对网络异常波动的安全保护措施，使用自我保护模式能使Eureka集群更加的健壮、稳定的运行
C. Eureka Server 在运行期间会去统计心跳失败比例在15分钟之内是否低于 85%，如果低于 85%，Eureka Server 即会进入自我保护机制
D. 在微服务架构下服务之间通常都是跨进程调用，网络通信往往会面临着各种问题，比如微服务状态正常，网络分区故障，导致此实例被注销

#### 正确答案
**C**

#### 简洁解析
1. A 正确：大量服务因网络误剔除会引发雪崩，自我保护就是解决该问题。
2. B 正确：应对网络抖动，避免误删健康服务，提升集群稳定性。
3. C 错误：阈值逻辑颠倒，15分钟内心跳失败比例**高于85%**才触发自我保护。
4. D 正确：网络不通但服务正常时，容易被错误注销实例，是自我保护要解决的场景。

#### 考点总结
Eureka自我保护触发条件：15分钟内失效实例比例超过85%，不再剔除服务实例。
